# User-facing strings (i18n catalog)

All user-facing copy in the site **chrome** — components, layouts, and pages —
lives in a single catalog, `src/i18n/en.ts`, and is referenced through
`strings` (from `src/i18n/index.ts`). A lint rule keeps literal copy out of
`.astro` markup.

This exists to support a Beeminder-wide rule: **user-facing text must not be
authored by AI.** Centralising copy gives humans one reviewable home for it and
makes accidental AI-written strings in `.astro` files fail CI instead of
shipping. (Post **content** is different — it comes from the CMS via
`posts.json` and is out of scope here.)

## Creating a NEW string — AI must not write the copy

**This is the core rule.** When you (an AI agent, or anyone scaffolding UI)
need a string that doesn't exist yet, **do not invent the wording.** Leave the
value blank and describe what's needed with `tk()`:

```ts
import { tk } from "./tk";
// ...
tagPage: {
  emptyState: tk("Message on the tag page when a tag has no posts"),
},
```

`tk(description)` renders the grep-able marker `[TK: ...]`, and CI (`check:i18n`)
**fails while any `tk(...)` stub remains** — so a human must replace it with the
real copy before the PR can merge. Reference the stub from `.astro` exactly like
any other string; the page will visibly show `[TK: ...]` until it's filled in.

Use `tk()` for static copy, interpolated copy (`emptyFor: (tag) => tk("...")`),
and HTML fragments alike — the human who fills it in chooses the final shape.

## Changing an existing string

Editing the wording of copy that's already written is a human's call too, but
mechanically: edit the entry in `src/i18n/en.ts`. Existing entries are:

- a plain string for static copy;
- a function for interpolation, e.g. `postCount: (count) => \`${count} posts\``;
- an HTML fragment when the copy contains inline links or emphasis (rendered
  with `set:html`).

Reference any entry in the `.astro` file:

```astro
---
import { strings } from "../i18n"; // ../../i18n from pages/*/**
---
<h3>{strings.archives.heading}</h3>
<p set:html={strings.sidebar.aboutLede} />
<img alt={strings.sidebar.dogfoodAlt} />
```

Never type literal copy directly in the `.astro` file.

## The lint guard

`eslint.config.js` runs `i18next/no-literal-string` (mode `jsx-only`) over
`.astro` files. It flags:

- JSX **text** content (`<h3>Archives</h3>` → error), and
- the user-facing **attributes** `title`, `alt`, `placeholder`, `aria-label`,
  `value` on native elements, plus literal string props on components.

Structural attributes (`class`, `href`, `src`, `id`, icon `name`, `Heading`,
etc.) and pure punctuation/numbers/acronyms are intentionally ignored. The
`.astro` `<script>`/`<style>` blocks are linted by `astro check`, not this rule.

**Known blind spot:** string literals defined in a component's frontmatter
(the `---` script block) are _not_ caught by this rule, because it only
inspects JSX. Build display strings in the template from `strings`, or assign
from `strings` in frontmatter (e.g. `const title = strings.listing.tagTitle(name)`),
rather than writing prose in frontmatter.

## Why not full i18n tooling?

The site ships a single English locale. `src/i18n/` is deliberately a thin
typed catalog with a locale barrel (`index.ts`) as the seam where locale
selection would go. The point today is **provenance and reviewability of
copy**, not translation.
